简单易懂理解Git的工作原理（2）：Index空间

发表于 2020-01-08 分类于 开发工具 ， Git

在上一篇文章中，我们解释了Git的数据模型。在这一篇文章，我们来看看Git另一个重要的方面：index。Index空间，也叫做stage空间，或者cache空间，集中了很多人对Git的误解。在这篇文章里，让我来尝试解开大家关于这方面的疑惑。
之前的我也对这部分有不少误解。我曾经以为，当你新checkout一个commit以后，你的index将是一片空白。然后你通过git add往index添加需要更新的文件。当你commit以后，你的新修改文件会被加入到Git数据库，然后你的index又会回到一片空白的状态。

我的理解实际上包含了几个错误的地方：

  • Index并不是临时存放新文件的地方
  • 新文件并不是在git commit的时候才被加入到Git的数据库。实际上在git add的时候就已经被加入到数据库里了。
  • Index包含的并不是当前commit和新commit的diff

其实严格来说，“把index.txt添加到index中”这个表述并不准确。因为实际上我们看到的是，当我们执行git add时，新的index.txt已经被作为一个新的blob对象添加到了Git数据库中了，而并不是被添加到index中。Git只是更新了index中这个文件的指针来指向新版本的blob对象。如果我们此时执行git commit，那么，Git将会基于index里面所指向的这些blob对象以及他们对应的路径来生成tree对象，最终生成commit对象（下图）。也就是说，实际上index空间是新的commit的一个写照（snapshot）。

解释了那么多，其实我想说的是，理解index的工作原理有助于我们自主去想解决的办法，而不是生搬硬套网上搜来的解决方法，是提高水平的一大重要途径。
